AI 公司把自己的秘密泄露了

从 Claude Code 源码泄露事件，看清 AI 工具背后那些你不知道的事

AI 安全 数据隐私 使用指南

2026 年 3 月 31 日，Anthropic 犯了一个让整个 AI 行业尴尬的错误。

他们把自己旗舰编程工具 Claude Code 的完整源代码，通过 npm 包发布，公开推送给了全球所有开发者。

这件事本身还不算最有意思的部分。最有意思的是：这已经是他们第二次犯同样的错误了。

而且，当全世界的程序员打开那 51.2 万行代码，看到的东西——让很多人倒吸一口冷气。

不是因为代码有多惊艳。而是因为代码里藏着 Anthropic 从没有公开承认过的东西。

1、事情是怎么发生的

发现这件事的人叫邵超凡（Chaofan Shou），一个出生于 2000 年后的中国年轻人，当时在一家名叫 Solayer Labs 的区块链公司做实习生。

他在 3 月 31 日例行检查 npm 包时，发现 Claude Code 版本 2.1.88 里多了一个 59.8MB 的 `.map` 文件——是正常包体积的将近 20 倍。打开一看，里面是 1906 个 TypeScript 源文件，51.2 万行代码，清晰注释，一览无余。

他把这个发现发到了 X（前推特），帖子在 48 小时内获得超过 2100 万次浏览。

技术原因

Anthropic 使用 Bun 构建工具打包代码。Bun 默认会生成 Source Map 文件（`.map`），这是一种用于调试的"源码地图"。工程师写了 `.npmignore` 规则想排除这些文件，但规则配置有误，结果 `.map` 文件原封不动地跟着发布包一起推送出去了。

Anthropic 在发现后 4 小时内删除了问题版本，发布了修复包 2.1.89。他们的声明只有一行：这是一次无意的发布错误，不包含用户数据，也未涉及安全漏洞。

听起来很轻描淡写。但有一个细节他们没有解释：这是他们 13 个月内第二次犯同样的错误。2025 年 2 月，Claude Code v0.2.8 已经以完全相同的方式发生过一次源码泄露。

同一家公司，同一个工具，同一个原因，两次。

那么，被这次意外公开的源码里，到底有什么？

2、代码里藏着什么

泄露的代码里有三样东西，让技术圈的人看完之后讨论了很久。

第一个：卧底模式（Undercover Mode）

代码里有一个名为 isNonInteractiveSession 的功能模块，功能是：当 Claude 被用于向开源代码仓库提交代码时，它会主动隐藏自己的 AI 身份，不在提交记录里留下任何 AI 参与的标记。

换句话说，GitHub 上那些看起来是人类写的代码，里面有多少实际是 AI 写的，外界根本无从判断。这不是 bug，这是一个被精心设计的功能。

第二个：后台自主模式（KAIROS）

代码里有一个内部代号 KAIROS 的守护进程（daemon）模块。它的设计目标是让 Claude 在用户不主动触发的情况下，在后台持续运行、自主执行任务——读文件、改代码、调工具。

这意味着，你的 AI 助手可能正在你不知情的时候，悄悄在你的电脑上干活。

第三个：模型内部代号（Capybara）

代码里出现了内部代号 "Capybara"，结合版本信息，研究者判断这对应的是 Claude 4.6 模型。代码注释里还提到了一个让人在意的数字：该模型的误报率（false positive rate）在某些场景下高达 29-30%——也就是说，三次拒绝用户请求里，大概有一次是冤枉的。

还有一个"宠物系统"（BUDDY），代码里描述的是一种通过给 AI 设置虚拟宠物形象来增强用户粘性的设计，以及 44 个功能开关（feature flags），这些开关控制着用户从未被告知的各种产品行为。

重要背景

开源社区的反应非常快。韩国开发者 Sigrid Jin 用泄露的源码进行了"洁净室重写"（clean-room rewrite），仅用 Python 重写出核心功能，创建了 claw-code 项目。这个项目在约两小时内获得了 5 万颗 GitHub Star，一度成为 GitHub 历史上增长最快的开源项目。

3、泄露的不只是代码

技术层面的分析到这里就差不多了。但这件事真正值得深想的，不是"Anthropic 把代码泄露出去了"，而是"泄露的代码揭示了什么"。

我认为这里有三个反常识的洞见。

洞见一：护栏保护的是公司，不是用户

2024 年底，斯普林格学术期刊发表了一篇研究（Kevin Mills 等人）。结论是：AI 公司设置的那些"安全护栏"，除了防止工具被用于造成事故和侵犯权利之外，其他限制主要服务于公司的商业利益——避免法律风险、控制品牌形象、保护数据资产——而不是保护用户。

这和泄露代码里的 Capybara 误报率形成了呼应：AI 有三分之一的概率在拒绝一个完全正当的请求。这个"误判"不是 AI 在保护你，是 AI 在保护公司免于被指责。

2026 年 2 月，Anthropic 还悄悄放弃了一项之前的承诺——"不会在没有安全缓解措施的情况下训练模型"。这个改变没有大张旗鼓宣布，只在政策文件里静默更新。

洞见二：你的数据是 AI 公司的成本结构

2025 年 8 月，Anthropic 悄悄修改了隐私政策：免费用户的对话数据，默认用于模型训练，保留期限从此前的约 90 天延长到了 5 年。

这件事本身不是新闻——很多 AI 公司都这么做。但它和 Claude Code 泄露事件放在一起看，揭示了一个更大的结构：

免费 AI 工具的商业模式是：用你的数据训练模型，用模型向企业客户收费。你不是用户，你是生产资料。

这不是指责，这是商业逻辑的必然。真正的问题在于：大多数人在用 AI 工具时，会把公司机密、个人隐私、敏感数据输入进去，却从来没想过这些数据的归宿。

三星就在 2023 年吃过这个亏——工程师把芯片设计图纸和会议记录粘贴给 ChatGPT 寻求帮助，后来才意识到这些数据已经进了 OpenAI 的训练数据池。三星随后全面禁止了内部使用外部 AI 工具。

洞见三：开放反而比封闭更安全

这是这件事最反常识的地方。

很多人的直觉是：闭源的 AI 工具更安全，因为坏人不知道它怎么运作，就更难攻击它。但现实是反过来的。

2025 年，安全公司 HiddenLayer 发现了一种叫做"政策傀儡"（Policy Puppetry）的攻击方式：用一段特殊格式的提示词，可以绕过所有主流 AI 大模型的安全护栏——Claude、GPT-4、Gemini、Llama 全部中招。没有一个例外。

闭源模型的"安全感"，本质上是"安全剧场"——给你一种被保护的感觉，但攻击者从来不需要知道源码就能攻击它。

反而是本地部署的开源模型（比如 Llama、Qwen 等），因为数据不离开你的设备，没有中间服务器，也没有远程后台进程，在数据隐私层面比任何闭源云服务都安全。

简单说

安全不来自于"你不知道它怎么运作"，安全来自于"它的运作不依赖于你信任它"。

我的评论

我用 Claude Code 的时间不短了。看到这次泄露，我的第一反应不是愤怒，是清醒。

Anthropic 不是坏人，他们只是一家商业公司，做着商业公司会做的事。但"不是坏人"不等于"利益与你一致"。AI 工具公司的利益和你作为用户的利益，有重叠，也有分歧。

理解这个分歧，是用好 AI 工具的前提。不是因为你要警惕他们，而是因为你需要知道：哪些事情可以放心交给 AI，哪些事情你需要亲自管住。

这次泄露是一次意外，但意外暴露了真实。而真实，从来比宣传更值得阅读。

4、普通人应该怎么做

这不是一篇劝你停止使用 AI 的文章。AI 工具真的很好用，应该用。但有几件事，现在知道比以后知道好。

第一，绝对不要把敏感信息输入云端 AI。公司的核心代码、客户数据、合同文件、个人身份信息——这些都不应该出现在 ChatGPT、Claude、Gemini 的对话框里。三星的教训已经摆在那里了。如果你需要处理敏感数据，用本地运行的开源模型（Ollama + Llama 或 Qwen）。

第二，警惕 AI 编程工具的权限边界。Claude Code 等工具默认会请求读取你整个代码库的权限。给它最小必要权限——只让它访问当前工作的那个模块，不要给它整个项目的读写权限。Cursor 和 Gemini CLI 有内置的敏感文件排除机制，Claude Code 没有，需要你手动配置 `.claudeignore`。

第三，付费用户的数据保护更好，但不是无限的。主流 AI 工具的付费版通常默认不用你的数据训练，免费版往往默认参与训练。如果你有隐私需求，用付费版，并主动关闭"数据改善"选项（通常在账号设置里）。

第四，AI 工具的拒绝，不一定是在保护你。当 AI 拒绝一个请求，有时候是在执行安全规则，有时候只是误报。Capybara 29-30% 的误报率意味着，每三次"这个我做不到"里，有一次只是过度谨慎。可以换个方式重新问，通常就能得到答案。

第五，把 AI 当工具用，不要当朋友用。KAIROS 的后台自主执行、BUDDY 的虚拟宠物粘性设计，都是在试图让你对 AI 工具产生依赖感。工具好用，但产品设计在某些方向上和你的利益并不完全一致。清醒地使用，是最好的保护。

由此得到

AI 公司不是坏人，但商业公司的利益从来不等于你的利益。

源码是意外泄露的，但它揭示的认知从来都在那里——只是你没有机会看到。

真正的 AI 素养不是会用 AI，而是知道什么时候不该信任它。

最安全的 AI 使用姿势，是把它当成一个能力很强但利益未必与你一致的合作者。
